# DAGON - Fishing Reservation Platform
> 낚시 예약 서비스를 중심으로, 사용자·파트너·관리자 권한 분리와 외부 API 연동을 고려해 설계한 Spring Boot · Vue 기반 웹 서비스

## 프로젝트 개요
> 낚시 상품 예약 서비스를 중심으로, 예약 흐름을 핵심 비즈니스로 설정하고 사용자·파트너·관리자 역할을 분리하여 설계한 웹서비스

일반 사용자는 낚시 상품을 조회하고 날짜, 인원, 옵션을 선택해 예약할 수 있으며, 파트너는 상품(낚시)과 조황정보를 관리하고, 관리자는 공지·이벤트·FAQ 등 서비스 운영을 위한 관리 기능을 수행

백엔드는 Spring Boot 기반의 계층형 아키텍처(Controller-Service-Repository)를 적용해 예약, 인증·권한, 외부 API 연동, 관리자 기능을 도메인 단위로 분리하여 구성했고, 프론트엔드는 Vue 기반으로 화면을 분리하고 API 연동 및 상태 관리를 담당

또한 물때·날씨와 같은 외부 공공데이터 API를 연동해 낚시 서비스 특성에 맞는 환경 정보를 제공하도록 설계

## 서비스 구조적 특징
> 예약 중심 비즈니스 흐름을 기준으로 도메인을 분리하고, 인증·권한·외부 API·관리자 기능을 독립적으로 구성한 구조가 특징

1. 예약 도메인을 중심으로 한 계층형 아키텍처를 적용하여 Controller-Service-Repository 역할을 명확히 분리하고 비즈니스 로직은 Service계층에 집중되도록 설계
2. 사용자 / 파트너 / 관리자 권한 분리 구조를 통해 API 경로와 인증 주체를 기준으로 접근 권한을 제어하고 프론트엔드 화면 구조 또한 사용자 유형에 따라 분리되어 동작
3. 물때·날씨 등 외부 공공데이터 API는 예약 도메인과 분리된 독립 도메인으로 구성해 외부 의존성을 최소화하고 향후 기능 확장 가능하도록 설계
4. 관리자 게시판(공지·이벤트·FAQ)과 같은 운영 기능을 서비스 기능과 분리해 실제 운영 환경을 고려한 구조 유지

→ 기능 간 결합도를 낮추고 확장·유지보수가 가능한 서비스 구조를 목표로 설계

## 내가 담당한 역할
> 협업 기준과 구조 문서화를 기반으로 시스템 구조 정리, 핵심 도메인 구현, 프론트엔드 UI 정리 담당
 
- 팀 공용 문서 구조를 설계하고, 요구사항·작업현황·회의내용을 문서화하여 공유 기준 정리
- 프로젝트 진행 중 구조·명칭·권한 기준이 혼란스러워지는 문제를 인식하고 2차 프로젝트 시작 전 **구조 및 명칭 가이드문서(DAGON 가이드)** 작성
- 사용자 유형(일반 사용자 / 파트너 / 관리자)을 기준으로 컨트롤러·패키지 구조를 정리하고, 분류가 어려운 영역은 legacy로 분리하여 관리
- 기존 코드 구조를 팀원들과 함께 검토하며 가이드 기준에 맞춰 리팩토링 및 main 브랜치 병합 조정
- 관리자 전용 게시판(공지사항·이벤트·FAQ) 도메인과 관리 API를 설계·구현
- 물때·날씨 외부 공공데이터 API를 연동하고, 서비스 도메인에 맞게 응답데이터를 가공 및 제공
- 프론트엔드에서 사용자 유형에 따른 헤더 구조를 설계·구현하고, 사이트 전반의 반응형 기준 수립하여 UI 정리
- 팀원들이 생성한 스타일을 분석·통합하여 공통 UI 스타일 기준을 정리하고 보조 UI 요소 구현
